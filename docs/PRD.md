## 二、产品概述

### 2.1 业务背景

随着企业信息化建设不断深入，企业内部及对外服务的应用系统日益增多（如Web应用、移动App、第三方系统等）。每个应用独立管理用户账号，导致以下问题：
- 用户需要在多个系统重复注册和登录，体验差
- 各系统用户数据分散，无法统一管理和审计
- 账号安全标准不统一，存在安全隐患
- 缺乏统一的操作审计和日志追溯能力

### 2.2 产品定位

用户中心管理系统是一套面向多应用场景的**统一身份认证与用户管理平台**，为各类应用系统提供集中的用户注册、登录、身份认证、令牌管理、应用绑定、操作审计、邮件通知及站点配置等基础能力。

**核心定位**：统一的用户身份与权限管理入口。

**职责边界**：
- ✅ **核心职责**：用户基础信息管理、认证授权、令牌生成与验证、审计日志、多应用绑定、邮件通知、站点配置
- ❌ **非核心职责**：具体业务数据存储（如订单、积分）、业务逻辑处理（如等级计算）、大规模用户行为分析

### 2.3 产品目标

| 目标 | 衡量指标 |
|------|----------|
| 实现统一身份认证 | 所有接入应用共享同一用户体系 |
| 支持JWT令牌颁发与验证 | API接口100%支持Token认证 |
| 支持OAuth2协议 | 第三方应用可通过标准OAuth2接入 |
| 多应用统一绑定与管理 | 支持≥50个应用同时接入 |
| 全链路操作审计 | 所有敏感操作100%记录 |
| 邮件通知能力 | 支持注册验证、密码重置等场景 |
| 站点配置动态化 | 配置变更无需重启服务 |

### 2.4 目标用户

| 用户角色 | 描述 | 核心诉求 |
|----------|------|----------|
| 普通用户 | 使用接入应用的最终用户 | 邮箱注册登录、个人信息管理、跨应用单点登录 |
| 应用开发者 | 接入用户中心的第三方应用开发者 | 标准化的接入文档、OAuth2/JWT接口、调试工具 |
| 系统管理员 | 管理用户中心的后台运营人员 | 用户管理、应用管理、配置管理、日志审计 |
| 企业管理员 | 企业内部的系统管理者 | 安全合规、操作可追溯、统一权限管控 |


## 三、用户场景

### 3.1 核心用户故事

**故事一：用户注册与登录**
> 作为一个普通用户，我希望能通过邮箱注册账号，设置密码后登录系统，以便访问接入用户中心的所有应用。

**故事二：单点登录（SSO）**
> 作为一个已登录的用户，当我从应用A跳转到应用B时，不需要重新输入用户名和密码，系统应自动识别我的登录状态。

**故事三：多应用绑定**
> 作为一个应用开发者，我希望将我的应用注册到用户中心，通过OAuth2协议让用户授权登录，以便复用用户中心的账号体系。

**故事四：密码找回**
> 作为一个忘记密码的用户，我希望能通过注册邮箱接收重置链接或验证码，安全地重置我的密码。

**故事五：操作审计**
> 作为一个系统管理员，我需要查看所有用户的操作日志，以便在发生安全事件时进行追溯和定责。

**故事六：账号注销**
> 作为一个不再使用本系统的用户，我希望能主动注销我的账号，系统将对我的个人身份信息（邮箱、昵称、头像）进行匿名化处理，仅保留脱敏后的操作审计记录以满足合规要求。

### 4.2 认证授权模块（JWT + OAuth2）

#### 4.2.1 对IdentityServer/OpenIddict的配置要求

> **架构决策**：本系统不自研JWT令牌服务。所有Token的颁发、验签、撤销逻辑统一由 **Duende IdentityServer**（或备选 **OpenIddict**）承载。签名密钥由其内置凭据管理机制（RSA密钥对/X.509证书）管理，通过 JWKS 端点（`/.well-known/openid-configuration/jwks`）对外发布公钥。本节定义对认证服务的配置要求，而非自研实现。

| 需求编号 | 需求描述 | 优先级 | 验收标准 |
|----------|----------|--------|----------|
| AT-001 | 登录成功后由IdentityServer颁发JWT Access Token | P0 | Token包含用户ID、邮箱、用户名、昵称、角色等声明 |
| AT-002 | Access Token有效期可配置 | P0 | 默认30分钟，可通过站点设置调整（上限60分钟） |
| AT-003 | 支持Refresh Token滚动刷新机制 | P0 | 每次刷新同时废弃旧Token并生成新Token；单次Refresh Token有效期7天，绝对生命周期上限30天（自首次颁发起算），超期后必须重新登录 |
| AT-004 | Token验签由IdentityServer中间件完成 | P0 | 无效/过期Token拒绝访问，返回401 |
| AT-005 | 登出时Refresh Token失效 | P1 | 登出操作将使当前Refresh Token失效；已颁发的Access Token将在其自然生命周期（默认30分钟）结束后无法换取新的Refresh Token，从而实现安全退出 |
| AT-006 | 签名密钥由IdentityServer凭据机制管理 | P0 | 签名密钥（RSA密钥对/X.509证书）由IdentityServer/OpenIddict内置机制管理；通过JWKS端点发布公钥，密钥轮换通过在JWKS中同时发布新旧kid实现，旧kid保留至所有存量Token自然过期后移除 |

**JWT载荷设计**（由IdentityServer按标准Claims映射）：
```json
{
  "sub": "用户ID",
  "email": "用户邮箱",
  "name": "用户名",
  "nickname": "昵称",
  "avatar": "头像URL",
  "roles": ["角色列表"],
  "iat": "签发时间",
  "exp": "过期时间",
  "jti": "Token唯一标识"
}
```

### 4.8 用户端页面（一期必选）

OAuth2 授权码流程要求用户中心必须托管以下页面以完成用户交互闭环。这些页面是一期交付的必要组成部分，**并非二期工作**：

| 需求编号 | 需求描述 | 优先级 | 验收标准 |
|----------|----------|--------|----------|
| PG-001 | 登录页 | P0 | 邮箱+密码登录表单；支持"忘记密码"和"注册"入口；OAuth2授权流程中未登录用户将被重定向至此页，登录成功后回跳至应用回调地址 |
| PG-002 | 注册页 | P0 | 邮箱+昵称+密码注册表单；注册成功后引导至登录页或直接完成登录并回跳 |
| PG-003 | 授权同意页 | P0 | 展示应用名称、图标、申请的作用域；用户可选择"同意"或"拒绝"；同意后建立绑定关系并回跳至应用回调地址；已登录且已绑定用户在后续授权请求中可跳过此页（静默授权） |
| PG-004 | 密码重置页 | P0 | 邮箱输入 → 验证码验证 → 新密码设置；验证码通过邮件发送 |
| PG-005 | 个人中心页 | P1 | 展示/编辑个人信息（昵称、头像）；查看已绑定应用列表并支持解绑；修改密码入口；注销账号入口 |

> **实现方式**：用户端页面为独立的 **React + Tailwind CSS** 单页应用（SPA），通过 HTTP API 与后端 ASP.NET Core 服务通信。OAuth2 授权码流程由 React 路由接管——未登录用户被重定向至登录页，授权同意页通过后端返回的应用信息渲染。IdentityServer/OpenIddict 仅作为认证服务提供 Token 端点，不依赖其默认 Razor UI。前端项目独立构建为静态资源，部署时可托管于 ASP.NET Core 的 `wwwroot` 或独立 CDN/静态站点。

## 六、数据需求

### 6.1 核心数据实体

> **说明**：以下仅列出用户中心系统自身维护的核心业务实体。IdentityServer/OpenIddict的Client配置（如RedirectUris、AllowedScopes等）将直接使用其内置的`Client`存储表，Client密钥哈希也由框架内置存储管理。应用注册时通过`Application`扩展表与框架Client表关联，避免重复造轮子。`AppUser.IsEmailVerified` 直接映射 ASP.NET Core Identity 内置的 `EmailConfirmed` 字段，PRD 中保留 `IsEmailVerified` 仅作为业务术语。`RefreshToken` 优先复用 Duende IdentityServer 原生 `PersistedGrants` 表（已内置支持滑动+绝对生命周期及 `RefreshTokenUsage.OneTimeOnly` 一次性轮换），技术评审时确认框架能力覆盖后不再自建表。

| 实体 | 说明 | 主要字段 |
|------|------|----------|
| AppUser | 用户信息 | Id、UserName（默认等于Email）、Email、Nickname、AvatarUrl、IsActive、IsDeleted、IsEmailVerified（映射Identity.EmailConfirmed）、IsAdmin、CreatedAt、LastLoginAt |
| Application | 应用注册扩展表（关联IdentityServer Client表） | Id、ClientId（关联IdentityServer Client）、AppName、Description、OwnerUserId、IsActive、CreatedAt |
| RefreshToken | 刷新令牌（优先复用Duende PersistedGrants，技术评审确认后不再自建） | Id、Token、UserId、ClientId、IssuedAt、ExpiresAt、AbsoluteExpiresAt、IsRevoked、ReplacedById |
| UserAppBinding | 用户-应用授权关系 | Id、UserId、AppId（关联Application）、BoundAt、LastAccessAt |
| AuditLog | 审计日志（用户注销时UserName和UserIp字段同步匿名化） | Id、UserId、UserName、UserIp、EventType、Description、BeforeValue、AfterValue、CreatedAt |
| SiteSetting | 站点配置 | Id、Key、Value（敏感项加密存储）、Group、DataType、IsSensitive、UpdatedAt、UpdatedBy |
| EmailTemplate | 邮件模板 | Id、Name、Subject、Body、Variables |

### 6.2 数据生命周期

| 数据类型 | 保留期限 | 清理策略 |
|----------|----------|----------|
| 用户数据 | 匿名化后永久保留审计骨架 | 注销时对PII字段匿名化：UserName和Email→`deleted_{GUID}`随机值、Nickname→"[已注销]"、AvatarUrl→置空；同步匿名化AuditLog中该用户历史记录的UserName→"[已注销]"、UserIp→"[REDACTED]"；不可逆处理后保留记录以满足审计合规 |
| 审计日志 | 180天（用户注销时即时匿名化） | 自动清理超期日志；用户注销时同步匿名化UserName和UserIp字段 |
| Refresh Token | 单次7天，绝对30天 | 过期自动失效；滚动刷新时旧Token标记撤销；禁用/注销时立即撤销 |
| 邮件发送记录 | 90天 | 自动清理 |


## 七、约束与假设

### 7.1 技术约束

| 约束项 | 说明 |
|--------|------|
| 后端框架 | ASP.NET Core 10 |
| 前端框架 | React 18+ + Tailwind CSS 4，独立 SPA 项目 |
| 缓存方案 | IMemoryCache（进程内缓存），不使用Redis |
| 认证框架 | ASP.NET Core Identity + JWT Bearer |
| Access Token 有效期上限 | 固定为 60 分钟（部署级约束，通过 `appsettings.json` 或环境变量设置，不在 SiteSetting 运行时配置表中出现） |
| OAuth2 / OpenID Connect服务 | **Duende IdentityServer** 或备选 **OpenIddict**（开源、MIT协议）。Duende IdentityServer v6+ 对商业用途有严格授权限制（公司年收入超过100万美金需购买许可证），最终选型将在技术方案评审时根据公司法务意见确定 |
| 邮件发送 | MailKit |
| 日志 | Serilog |
| ORM | Entity Framework Core |

### 7.2 假设条件

1. 部署环境为单机或小规模集群（<10个实例），IMemoryCache可满足缓存需求
2. 接入应用数量初始≤50个
3. 用户规模初始≤100万
4. SMTP邮件服务器由运维团队提供
5. 数据库由DBA团队负责维护和备份
6. 系统接受JWT撤销的最终一致性（最长为已配置的 Access Token 有效期，≤60 分钟）；如业务方有更高实时性要求，需单独申请启用Redis分布式缓存以实现即时Token撤销
7. Refresh Token因存储在数据库中，撤销为即时生效，不受IMemoryCache多实例不一致影响

## 九、风险与应对

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| IMemoryCache缓存不一致 | 多实例部署时缓存数据不一致 | 中 | 设置短过期时间，必要时切换Redis |
